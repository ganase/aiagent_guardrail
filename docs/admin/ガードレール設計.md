# ガードレール設計（v0.2）

## 基本方針

標準インストーラーにより、標準設定、ガードレール、テンプレート、許可済みパッケージリスト、バージョン管理を配布します。

## パッケージ導入ポリシー（deny-list-only 設計）

「deny リストに載っているものだけブロック、未知は通す」という設計です。allowlist のメンテナンス負荷を排除しつつ、既知悪性パッケージと typosquat を確実に防ぎます。

| 区分 | 判定 | hook の動作 |
|---|---|---|
| deny リスト該当 | 明示的禁止・既知悪性 | exit 2 でブロック。stderr に理由と相談先 |
| typosquat 疑い | allowlist の既知名と Levenshtein 距離 1〜2 | exit 2 でブロック。stderr に類似パッケージ名と警告 |
| それ以外（allow / review リスト該当・未知） | すべて自動通過 | exit 0 + JSON `permissionDecision: allow` |

**typosquat 検出**: 抽出したパッケージ名と allowlist の allow / review エントリ（名前が既知の基準点として機能）を Levenshtein 距離で比較。距離 1〜2 の場合は deny（exit 2）。allow / review エントリはインストール審査対象ではなく、typosquat 検出の参照名一覧として機能します。

## ガードレール一覧

| ガードレール | 実装 | 主な対象リスク | 限界 |
|---|---|---|---|
| Sandbox | Codex Windows sandbox / Claude 権限制御 | ローカル破損、本番影響、危険コマンド | 外部送信や誤承認は単独では防げない |
| Permission / Approval | Claude permissions / Codex approval | 危険操作、外部送信、本番影響 | 承認疲れ・誤承認は残る |
| Hook (fail-closed) | `aiagent_guardrail_check.py` + `run_guardrail_hook.cmd` | pip/npm/uv/poetry/conda、危険コマンド、ランタイム導入、機密ファイル読取、PreToolUse Read/Edit/Write ファイルパスチェック、UserPromptSubmit credential スキャン（ブロック）、PostToolUse credential スキャン（警告） | **文字列マッチのため原理的に回避可能**（python -c、スクリプト経由等）。推移的依存は検査対象外 |
| deny-list-only ポリシー | `package_allowlist.json` | サプライチェーン、既知悪性ライブラリ、typosquat | deny リスト外の未知パッケージは自動通過。バージョン範囲は実行時には検証していない |
| deny / ignore | Claude permissions / AGENTS.md | Secret 読取、機密ファイル読取 | ユーザーが別経路で渡すと防げない |
| 設定ハッシュ（改ざん検知） | `installed_hashes.csv` + 起動時 SHA256 検証（policy/allowlist/runtime_policy の各 JSON、hook 本体、run_guardrail_hook.cmd、テンプレートを対象。cli/claude-hook 両モードで検証） | 設定改変、Hook本体の改ざん、バージョン差異 | **非 admin 導入時はユーザーが設定ファイルを改変可能**。hash ファイル自体も書き換え可能（ACL 保護が前提） |
| blocked_paths | `aiagent_guardrail_check.py` コード実装 | Bash 経由の .env / secrets / 秘密鍵読取 | **完全な防御ではない**。python -c open('.env')、スクリプト経由は防げない |

## 既知の限界（実証済み）

実証済みの穴・恒久対策の詳細一覧は [既知の限界.md](既知の限界.md) を正とします（本設計書と重複させないため、要約表はそちらに一本化しています）。

## Claude Code

managed settings と PreToolUse hook を使用します。hook コマンドは `run_guardrail_hook.cmd` 経由（fail-closed ラッパー）。パッケージ導入の ask は hook の JSON に一元化し、managed-settings からは削除済みです（二重摩擦解消）。

## Codex

Windows sandbox、`config.toml`、`requirements.toml`、`AGENTS.md`、`ai-pip` / `ai-npm` を併用します。

**重要：Codex には Hook 機構が存在しない**：AGENTS.md は助言であり、ai-pip/ai-npm は任意利用です。素の pip/npm コマンドを Codex が実行する場合は制限されません。恒久対策は docs/必要な仕組み一覧.md のロードマップ参照。

**コスト制御（R9/R14）の限界**：Claude Code で実装した Hook 呼び出し回数制限（50/100回ブロック）は Codex では発動しません。Codex のコスト制御手段は `~/.codex/config.toml` の `approval_policy` 設定のみです。

| approval_policy 値 | 挙動 |
|---|---|
| `manual` | 全操作で人間確認（推奨） |
| `on-failure` | 失敗時のみ確認 |
| `auto` | 全操作を自動承認（**非推奨**：コスト暴走リスク最大） |

`check_status.ps1` が設定値を確認し、`auto` または未設定の場合に警告します。ただし設定値の強制適用はできません（利用者が書き換え可能）。詳細は [既知の限界.md §1](既知の限界.md) 参照。
