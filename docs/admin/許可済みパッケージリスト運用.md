# 許可済みパッケージリスト運用（v0.2）

## 方針

- default policy は deny。
- allow / review / deny の 3 区分で管理する。
- **未知パッケージは deny ではなく ask（人間承認）**。承認フロー → allowlist 反映、という運用で摩擦を減らす。
- review 扱いはガバナンスチームまたは管理者確認後に利用する。

## 3 層ポリシーと hook の動作

3層（allow / review・未知 / deny）と hook の判定・managed-settingsとの関係は [ガードレール設計.md](ガードレール設計.md) の「パッケージ導入ポリシー（3層設計）」を正とします。本文書は、その運用（フロー・ログ・更新手順）に絞って記載します。

## 未知パッケージのフロー

```text
未知パッケージを検知
  ↓
hook が permissionDecision: ask を返す
  ↓
ユーザーが承認ダイアログで判断
  ↓
[承認] → 実行される（logs/package_decisions.csv に記録）
  ↓
ガバナンスチームが定期レビュー（週次/月次推奨）
  ↓
allowlist 反映（allow / review / deny に分類）
  ↓
標準インストーラーまたは標準モジュール更新で配布
```

## 導入ログ

**場所**: `{InstallRoot}/logs/package_decisions.csv`

**形式**: `timestamp,user,ecosystem,package,decision,command`

| カラム | 内容 |
|---|---|
| timestamp | UTC ISO8601 |
| user | OS ユーザー名 |
| ecosystem | python / javascript |
| package | パッケージ名 |
| decision | allow / ask / ask(review) / deny / deny(typosquat) |
| command | 実行コマンド（先頭 500 文字） |

ログ書込失敗は無音で無視します（ベストエフォート）。判定には影響しません。

## 更新フロー

| 手順 | 内容 |
|---|---|
| 1 | 利用者が用途、パッケージ名、バージョン、理由を申請する |
| 2 | AIガバナンスチームがリスク確認する |
| 3 | allow / review / deny を決定する |
| 4 | `package_allowlist.json` を更新する |
| 5 | `validate_allowlist.py` で検証する |
| 6 | 標準インストーラーまたは標準モジュール更新で配布する |

## リスク確認観点

| 観点 | 内容 |
|---|---|
| 既知脆弱性 | OSV、pip-audit、npm audit 等で確認 |
| メンテナンス状況 | 更新日、リリース頻度、メンテナ数 |
| タイポスクワッティング | 有名パッケージ名との Levenshtein 距離 ≤ 2 で自動検知 |
| 外部通信 | HTTP 通信・認証・ブラウザ操作の有無 |
| 依存関係 | 依存パッケージ数、依存先の安全性（推移的依存は手動確認） |
| ライセンス | 社内利用可能なライセンスか |

## 重要: ログの定期レビュー

ask で承認されたパッケージを定期的（週次/月次）に `package_decisions.csv` で確認し、繰り返し使われているものは allowlist に反映してください。ask の放置は承認疲れにつながります。
