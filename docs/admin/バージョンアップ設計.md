# バージョンアップ設計

## 目的

AI Agent本体、標準インストーラー、標準設定、Hook、テンプレートのバージョン差異を防ぎ、最新のリスク対策を利用者へ届ける。

## 1. 管理対象

| 管理対象 | 例 |
|---|---|
| **AI Agent本体** | Codex CLI、Claude Code 等 |
| **標準インストーラー** | 導入スクリプト、配布パッケージ |
| **標準設定** | 設定ファイル、権限、ログ、接続先 |
| **プロジェクト指示** | AGENTS.md、CLAUDE.md |
| **ガードレール** | Hook、deny / ignore、許可リスト、Sandbox設定、インストールHook / ラッパー |
| **テンプレート** | README、.gitignore、サンプル、Security Check手順 |
| **許可済みパッケージリスト** | pip/npm等のライブラリ導入可否、許可バージョン、確認日、管理者 |

## 2. バージョン管理の考え方

| 種別 | 変更例 |
|---|---|
| **メジャー** | 利用ルール、設定形式、配布方式が大きく変わる。 |
| **マイナー** | Hook、テンプレート、許可リスト、許可済みパッケージリスト、ログ項目を追加する。 |
| **パッチ** | 誤記修正、軽微な設定修正、不具合修正。 |

AI Agent本体のバージョンと、標準インストーラーのバージョンは分けて管理する。

## 3. 更新フロー

| フェーズ | 内容 |
|---|---|
| **1. 変更把握** | Agent本体、設定、Hook、テンプレートの更新要否を確認する。 |
| **2. 検証** | サンプル、既存テンプレート、設定ハッシュ、ログ出力を確認する。 |
| **3. パッケージ作成** | 新しい標準インストーラーと標準設定を作成する。 |
| **4. 先行配布** | 一部利用者で動作確認する。 |
| **5. 全体配布** | 対象者へ更新案内し、標準インストーラーで更新する。 |
| **6. 更新確認** | ハートビートと対象者リストを突合する。 |
| **7. 改善** | 問い合わせ、失敗事例、ヒヤリハットを反映する。 |

## 4. 更新確認項目

| 確認項目 | 内容 |
|---|---|
| **Agent本体バージョン** | 指定された標準版か。 |
| **標準インストーラー版** | 最新の配布パッケージか。 |
| **標準設定版** | 許可リスト、Hook、テンプレート、許可済みパッケージリストが最新か。 |
| **設定ハッシュ** | 標準設定から改変されていないか。 |
| **最終ハートビート** | 最近確認できているか。 |
| **旧版利用期限** | 期限を過ぎた旧版が残っていないか。 |

## 5. ロールバック

問題発生時に旧バージョンへ戻せるよう、前版パッケージ、旧設定、戻し手順を保管する。重大なセキュリティ修正を含む場合は、旧版利用期限を短くする。


## 6. 許可済みパッケージリストの更新

| 項目 | 内容 |
|---|---|
| **更新単位** | 標準モジュール配布時に更新する。緊急脆弱性がある場合は臨時更新する。 |
| **管理項目** | エコシステム、パッケージ名、許可バージョン、状態、理由、確認日、管理者を記録する。 |
| **状態** | allow / review / deny の3段階で管理する。 |
| **確認観点** | 既知脆弱性、更新状況、利用実績、依存関係、類似名、外部通信・コマンド実行の有無を確認する。 |
| **配布** | 標準インストーラーに同梱し、設定ハッシュとバージョンで改変・差異を確認する。 |
