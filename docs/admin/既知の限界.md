# 既知の限界（v0.2 時点）

本ガードレールが対処できない・対処が困難な既知の穴を正直に記します。対策なしに「安全」と思わないようにしてください。

## 1. Codex 側には強制力のある enforcement が存在しない

AGENTS.md はモデルへの助言であり、強制力はありません。`ai-pip` / `ai-npm` は任意利用であり、Codex が素の `pip install` / `npm install` を直接実行した場合、hook は発動しません。

**コスト制御でも同様に適用外**：Claude Code で実装している Hook 呼び出し回数制限（直近60分 50回→警告、100回→ブロック）は Codex には発動しません。Codex のコスト制御手段は `~/.codex/config.toml` の `approval_policy` 設定のみです。

| approval_policy 値 | 挙動 | 推奨 |
|---|---|---|
| `manual` | 全操作で人間確認 | 推奨 |
| `on-failure` | 失敗時のみ確認 | 許容（コスト監視と併用） |
| `auto` | 全操作を自動承認 | 非推奨（コスト暴走リスク最大） |

`check_status.ps1` が `approval_policy` の設定値を確認し、`auto` または未設定の場合に警告します。ただし、設定値の強制適用はできません（利用者が意図的に `auto` に戻せる）。

**恒久対策**: ネットワーク境界（社内 PyPI/npm ミラー + egress プロキシ）による allowlist 強制。Codex の `approval_policy` を `manual` に設定し、定期的に `check_status.ps1` で確認する。

## 2. 文字列マッチの原理的回避可能性

hook は Bash コマンドの文字列に対するパターンマッチです。以下のような手法で容易に回避できます。

```python
# hook は発動しない
python -c "__import__('pip').main(['install','evil-pkg'])"

# スクリプトファイル経由
python install_something.py

# 環境変数経由
PACKAGE=evil-pkg pip install $PACKAGE

# base64 エンコード
pwsh -enc <base64 of pip install evil-pkg>
```

**恒久対策**: ネットワーク境界が唯一の呼び出し方法に依存しない防御。

## 3. 推移的依存の未検査

`pip install pandas` を許可した場合、pandas が依存している numpy、python-dateutil 等は個別にチェックされません。悪意ある依存関係が混入しても、直接指定でなければ検知できません。

**恒久対策**: `pip-audit` / `npm audit` による定期スキャン。lock ファイルの管理。

## 4. 非 admin 導入時の改変可能性

管理者権限なしでインストールした場合:
- ACL 保護が設定されません
- ユーザーが `guardrail_policy.json`、`package_allowlist.json`、`aiagent_guardrail_check.py`、`installed_hashes.csv` を自由に書き換え可能です（改ざん検知の対象ファイルを増やしても、ACL がなければ検知データ自体も一緒に書き換えられるため無力化できます）
- `AIAGENT_GUARDRAIL_HOME` 環境変数（CLI モード）を、`installed_hashes.csv` が存在しない別ディレクトリに差し替えるだけで、改ざん検知は「開発環境」として無条件通過します

**恒久対策**: 管理者による正式導入（ACL 保護付き）を徹底する。非 admin 導入は評価用途限定と明記する。`check_status.ps1` の「ACL protection / install level」欄で導入状態を確認できる。

## 5. 承認疲れ（ask の乱発）

未知パッケージが多い環境では `ask` ダイアログが頻出し、人間が中身を確認せずに承認するリスクがあります。

**軽減策**: `logs/package_decisions.csv` の定期レビューで ask の多いパッケージを allowlist に取り込む。allowlist の整備で ask を減らす。

## 6. ユーザー自身のターミナルは対象外

Claude Code が提案した Bash コマンドは hook でチェックされますが、ユーザーがターミナルで直接 `pip install X` を実行しても hook は発動しません。

**恒久対策**: ネットワーク境界。端末管理ポリシー。

## 7. installed_hashes.csv 自体の改ざん

改ざん検知は `installed_hashes.csv` の内容を正として照合します。ACL 保護がない環境（非 admin 導入時）では `installed_hashes.csv` 自体を書き換えることで改ざん検知を無効化できます。検証対象は `guardrail_policy.json` / `package_allowlist.json` に加え、hook 本体（`aiagent_guardrail_check.py`）・`run_guardrail_hook.cmd`・テンプレートにも拡張済みですが、この限界自体は解消していません（ACL 保護が唯一の前提）。

**恒久対策**: ACL 保護付きの管理者導入。

## 8. バージョン範囲の実行時未検証

`allowed_versions` フィールド（例: `>=2.0,<3.0`）は実行時に「インストールしようとしているバージョンが範囲内か」の検証を行いません。`pandas==1.0.0`（古い脆弱バージョン）も `pandas==2.0.0` も同じく allow になります。なお `expires_at`（審査有効期限）は実行時に検証されており、期限切れの allow エントリは自動的に ask（再審査待ち）へ降格します。ただしこれは「審査した日から一定期間が経ったら人間に再確認させる」仕組みであり、バージョン範囲そのものの検証ではありません。

**恒久対策**: `pip-audit` による定期スキャン。

## 9. ハッシュファイル非存在時の動作

`installed_hashes.csv` が存在しない場合（開発環境・インストール直後等）は警告なしに通過します。改ざんとハッシュファイル不在の区別ができません。

## まとめ: ガードレールの位置づけ

本ガードレールは「完全なセキュリティ」ではなく「善意の AI エージェントに対する摩擦と監査ログの提供」と位置づけてください。悪意ある内部者や高度な攻撃者に対しては有効ではありません。ネットワーク境界・端末管理・教育・定期レビューとの組み合わせが必要です。
